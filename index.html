Common Windows issues: solved, free.

A working sample of our public knowledge base. These resolve the majority of day-to-day office tickets. Deeper, fleet-wide, or recurring cases are covered by the subscription.

The most common cause is a corrupted update cache. Resolve it without third-party tools:

  1. Open Settings → Windows Update → Pause updates for 1 week, then resume; this often clears a wedged queue.
  2. If it persists, run net stop wuauserv and net stop bits in an elevated terminal, rename C:\Windows\SoftwareDistribution to SoftwareDistribution.old, then restart both services.
  3. Run sfc /scannow followed by DISM /Online /Cleanup-Image /RestoreHealth to repair system files.

Fleet-wide on a managed network? Subscribers get a scripted rollout and WSUS/Intune review instead of per-machine fixes.

  1. Open Settings → Bluetooth & devices → Printers & scanners, select the printer, and clear Use Printer Offline if it's ticked.
  2. Restart the Print Spooler: net stop spooler then net start spooler in an elevated terminal.
  3. Confirm the printer's IP hasn't changed (DHCP lease). Re-add it by IP and set a DHCP reservation on the router to stop it recurring.

Whole office hitting this after every reboot? It's usually a DHCP or print-server config, so we fix the root cause under a subscription.

  1. Open Task Manager → Startup apps and disable anything non-essential (updaters, vendor trays, chat clients set to auto-launch).
  2. Check Settings → System → Storage; a drive over ~90% full slows boot and paging dramatically.
  3. On older fleets, confirm the OS is on an SSD; a single HDD-to-SSD swap is usually the largest single gain.

For a fleet baseline (boot times, startup policy, image cleanup) across many machines, that's a subscription engagement.

  1. If the device used a digital license, run Settings → System → Activation → Troubleshoot; the activation troubleshooter can re-bind after a motherboard or major-component change.
  2. For volume-licensed offices, confirm the KMS host is reachable and the client is using the correct GVLK.
  3. Never enter a key from an untrusted "activation" tool; that's how offices end up on non-genuine builds.
